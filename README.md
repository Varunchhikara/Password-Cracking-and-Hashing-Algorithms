# Password Cracking & Hashing Algorithms

**Objective:** Analyze and implement brute-force password cracking tools while
understanding hash functions and salting techniques.

**Tools:** Python 3, John the Ripper, Hashcat (Kali Linux)

**Skills demonstrated:** Hashing fundamentals, salting, brute-force attacks,
dictionary/rule-based attacks, key-derivation functions (bcrypt), and how
real-world cracking tools apply these concepts at scale.

## Scope & Ethics

Every hash cracked in this project targets **passwords generated locally by
this project's own scripts** — never real accounts, leaked breach data, or
systems the author doesn't own. The goal is to understand *why* password
cracking works, not to build a tool for unauthorized access. Run only against
systems/data you own or are explicitly authorized to test.

## Part 1 — Understanding Hashing (`01_hashing_basics.py`)

Covers, with runnable demonstrations:

- **Determinism** — same input always produces the same hash, which is what
  makes precomputed *rainbow tables* possible against unsalted hashes.
- **Avalanche effect** — a one-character input change produces a completely
  different output, a core property of a well-designed hash function.
- **Salting** — a random per-user value mixed into the hash before hashing.
  Two users with the identical password get different stored hashes, which
  defeats rainbow tables and prevents cross-user password-reuse leakage.
- **Speed** — MD5/SHA1/SHA256 compute millions of hashes/sec on a single
  CPU core in pure Python; on a GPU with Hashcat that reaches billions/sec.
  This single fact is *why* fast hash functions are unsafe for passwords no
  matter how "the hash looks random."

## Part 2 — Brute Force (`02_bruteforce_cracker.py`)

A from-scratch brute-force cracker that enumerates every possible string up
to a given length/charset and hashes each one until it matches the target.
Demonstrates:

- Exponential keyspace growth (`charset_size ^ length`), and why brute force
  is only practical against short/low-complexity passwords.
- That salting does **not** slow down cracking a single hash — it only
  prevents attackers from reusing work across many hashes. The real defenses
  against brute force are password *length/complexity* and a *slow* hashing
  algorithm (see Part 4).

## Part 3 — Dictionary & Rule-Based Attacks (`03_dictionary_cracker.py`)

Most real-world leaked passwords aren't random — they're dictionary words
with predictable mutations (`dragon` → `Dragon123!`). This script:

- Loads a wordlist and tries each word as-is (mostly misses).
- Applies **mangling rules** (leetspeak substitution, capitalization, common
  numeric/symbol suffixes) — mirroring what John the Ripper's rule engine
  and Hashcat's `-r` rule files do — and cracks the same "complex-looking"
  password in a handful of attempts instead of billions.

## Part 4 — Why Slow Hashing Matters (`04_slow_hash_demo.py`)

Benchmarks raw SHA256 against **bcrypt**, which has a tunable *work factor*.
On this environment, SHA256 was roughly **590,000× faster** than
bcrypt(cost=12) per hash — the entire reason OWASP/NIST mandate
bcrypt/scrypt/argon2id for password storage instead of "just add a salt to
SHA256." Also shows correct bcrypt usage (`hashpw` / `checkpw`), where the
salt is embedded directly in the stored hash string.

## Part 5 — Real Tools: John the Ripper & Hashcat

`05_generate_test_hashes.py` produces sample hash files in the exact formats
these tools expect:

| File | Format | John | Hashcat mode |
|---|---|---|---|
| `hashes_md5.txt` | Raw MD5 | `--format=Raw-MD5` | `-m 0` |
| `hashes_sha256.txt` | Raw SHA256 | `--format=Raw-SHA256` | `-m 1400` |
| `shadow_style.txt` | Linux SHA-512 crypt (`$6$`) | `--format=sha512crypt` | `-m 1800` |
| `hashes_bcrypt.txt` | bcrypt | `--format=bcrypt` | `-m 3200` |

`john_hashcat_cheatsheet.sh` walks through:
1. Dictionary attacks against each format
2. Rule-based/mangled attacks (John `--rules`, Hashcat `best64.rule`)
3. Mask/brute-force attacks (`hashcat -a 3 ?l?l?l?l`, John `--incremental`)
4. Scaling up to `rockyou.txt`, the standard 14M-password real-world wordlist
5. Verifying results against the generated answer key

## Key Findings / Takeaways

1. **Unsalted hashes are broken by design at scale** — not because the
   underlying algorithm is weak, but because identical inputs produce
   identical outputs, enabling rainbow tables and cross-user leakage.
2. **Salting fixes reuse-of-work attacks, not brute-force speed.** A salted
   MD5 hash is exactly as fast to crack individually as an unsalted one.
3. **Dictionary + rule-based attacks beat brute force in practice** because
   humans don't pick random passwords — they pick words with predictable
   mutations, which is exactly what John/Hashcat's default modes target.
4. **Algorithm choice for password storage matters more than "salt or not"**
   — bcrypt/scrypt/argon2id's tunable work factor is what actually raises
   attacker cost per guess by orders of magnitude, which raw SHA-family
   hashes cannot provide regardless of salting.
5. **GPUs change the threat model entirely** — the same MD5/SHA256 speeds
   that look "fine" on a CPU in Python become billions/sec on consumer GPU
   hardware via Hashcat, which is why length + slow-KDF, not just
   complexity rules, is the real mitigation.

## Mitigations (for anyone designing a real auth system)

- Never store plaintext or reversibly-encrypted passwords.
- Use bcrypt, scrypt, or Argon2id with an appropriately tuned cost factor —
  never raw MD5/SHA1/SHA256/SHA512, salted or not.
- Enforce meaningful minimum length (12+ chars) over arbitrary complexity
  rules that push users toward predictable patterns (`Password1!`).
- Rate-limit and lock out authentication attempts to blunt online
  brute-force/credential-stuffing even before offline cracking is relevant.
- Check new passwords against known-breached password lists (e.g. HaveIBeenPwned's
  Pwned Passwords API) to reject already-compromised passwords at signup.
